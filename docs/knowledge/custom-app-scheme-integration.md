# Custom App Scheme Integration in Chromium-Backed CBF

## Scope

This note captures what we learned while replacing `file://`-based UI loading in
`examples/simpleapp` with a Chrome-backend-specific `app://` custom scheme.

The target behavior was:

- load embedded UI assets from `app://simpleapp/...`
- keep public `cbf` API unchanged
- implement the feature in `cbf-chrome` and Chromium bridge code only
- support top-level document loads and same-origin subresource fetches

## What Was Confirmed

### 1. A runtime responder alone is not enough

Implementing a custom URL loader that returns bytes for `app://...` requests is
not sufficient by itself.

Without launch-time scheme classification, Chromium can fetch the resource but
does not treat the scheme as a normal web origin. In practice this caused:

- the HTML source to be shown as plain text instead of being rendered
- console errors such as:
  - `Unsafe attempt to load URL app://simpleapp/ui.html from frame with URL app://simpleapp/ui.html`
- network status like `(blocked:origin)`

The important distinction is:

- runtime registration answers requests for `scheme + host`
- launch-time registration teaches Chromium what the scheme *is*

Both are required.

### 2. Custom schemes must be registered as first-class web schemes at launch

To make `app://simpleapp/...` behave like a normal same-origin web document, the
scheme name must be declared before Chromium finishes startup.

The working configuration was:

- add custom scheme names to `standard_schemes`
- add custom scheme names to `secure_schemes`
- add custom scheme names to `cors_enabled_schemes`
- register the scheme as web-safe in `ChildProcessSecurityPolicy`
- register the scheme in the renderer as:
  - supporting Fetch API
  - allowed for referrer
- register the scheme with `url::AddWebStorageScheme(...)`

This was wired through a launch switch (`--cbf-custom-schemes=...`) generated by
`cbf-chrome` startup code from `ChromiumBackendOptions`.

### 3. The loader response head must look like a real document response

Even after scheme classification was added, the custom loader needed to build a
proper `URLResponseHead` for document loads.

The working response path included:

- parsing `Content-Type` into MIME type and charset separately
- setting `response->mime_type` to the parsed MIME type, not the raw header
- setting `response->charset` when present
- populating `content_length`
- providing a consistent `LoadTimingInfo`
- calling `OnReceiveResponse(...)` before `OnComplete(...)`

Returning only body bytes or an incomplete response head was not reliable enough
for HTML document rendering.

### 4. Same-origin subresource tests exposed an unrelated non-cryptographic IPC issue

Top-level `app://` navigation started working before same-origin subresource
fetch tests were stable.

The browser test crash was a bad Mojo message on
`FrameHost::SubresourceResponseStarted`. That path is effectively SSL-oriented:
`RenderFrameHostImpl::SubresourceResponseStarted(...)` forwards to
`SSLManager::DidStartResourceResponse(...)`, and the SSL manager immediately
returns for non-cryptographic schemes.

For `app://`, sending that IPC provided no value and destabilized the test.

The fix was to suppress `SubresourceResponseStarted` notifications in
`RenderFrameImpl` for non-cryptographic subresource URLs.

## Architecture Notes

The final split that worked well was:

- launch time:
  - declare which scheme names Chromium must treat as web-capable origins
- runtime:
  - register which `scheme + host` pairs CBF owns
  - forward each request to the host app
  - let the host respond with body, MIME type, CSP, and CORS headers

This preserved layer boundaries:

- `cbf` stayed browser-generic
- `cbf-chrome` exposed Chrome-specific startup and raw event/command support
- Chromium bridge code remained responsible for loader behavior and browser-side
  security policy integration

## Practical Guidance

If CBF adds another custom scheme in the future, follow this sequence:

1. Add the scheme name to launch-time Chromium registration.
2. Keep request ownership runtime-scoped to `scheme + host`.
3. Build a complete `URLResponseHead`, not just a body payload.
4. Treat non-cryptographic schemes carefully in browser/renderer security and
   notification paths.

If the symptom is "HTML source is displayed instead of rendered", check scheme
classification first, not just MIME type handling.

If the symptom is a same-origin browser test crash for `app://` subresources,
inspect renderer-to-browser notifications that assume HTTPS/TLS semantics.

## Validation Signals

The integration was considered healthy when all of the following passed:

- `simpleapp` rendered `app://simpleapp/ui.html` as HTML
- exact-origin IPC allowlisting worked with `app://simpleapp`
- top-level browser test for `app://` navigation passed
- same-origin subresource fetch browser test passed
